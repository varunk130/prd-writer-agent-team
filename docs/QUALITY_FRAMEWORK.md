# Quality Framework: PRD Scoring Engine

## Overview

Every PRD generated by the agent team is evaluated across 5 independent dimensions. This framework ensures quality is measurable, comparable, and improvable — not subjective.

## Why Dimensional Scoring?

A single "quality score" hides critical information. Consider:
- An agent might score 95 on Technical Depth but 72 on User Stories
- That's a very different document than one scoring 83 on both

Dimensional scoring reveals the **shape** of quality, not just its magnitude.

## The 5 Dimensions

### 1. Completeness (0-100)

**Definition**: Are all expected PRD sections present and substantive?

**Evaluation criteria**:
- Executive Summary present and concise
- Problem Statement with quantified impact
- User Research Summary with data points
- Goals and Non-Goals clearly delineated
- Proposed Solution with architecture description
- User Stories with acceptance criteria
- Technical Considerations addressing constraints
- Success Metrics that are measurable
- Dependencies and Risks identified
- Timeline with milestones
- Open Questions documented
- Competitive Analysis included

**Scoring**: Each section contributes proportionally. Missing sections receive 0. Stub sections (< 50 words) are penalized.

### 2. Specificity (0-100)

**Definition**: Does the PRD use concrete numbers, standards, and frameworks rather than vague language?

**Evaluation criteria**:
- Quantified metrics (e.g., "reduce from 8 hours to 45 minutes" vs. "improve efficiency")
- Named standards and frameworks (e.g., "HL7 FHIR" vs. "industry standards")
- Specific user segments (e.g., "buy-side investment analysts at firms with AUM > $5B" vs. "financial professionals")
- Concrete acceptance criteria (e.g., "API response < 200ms at p95" vs. "fast response times")

**Anti-patterns detected**:
- "Various stakeholders" → Who specifically?
- "Improved performance" → By how much?
- "Industry best practices" → Which ones?
- "Scalable architecture" → To what scale?

### 3. User Stories (0-100)

**Definition**: Are user stories well-formed with acceptance criteria?

**Evaluation criteria**:
- Follows "As a [role], I want [capability], so that [benefit]" format
- Role is a specific persona, not generic
- Capability is actionable and testable
- Benefit connects to a measurable outcome
- Acceptance criteria are included
- Edge cases are considered
- Priority is assigned (P0/P1/P2)

### 4. Technical Depth (0-100)

**Definition**: Does the technical section address real architecture, integration, and constraint concerns?

**Evaluation criteria**:
- System architecture described (not just buzzwords)
- Integration points identified with specific APIs/protocols
- Data model or schema considerations
- Performance requirements quantified
- Security and compliance requirements specific to the domain
- Infrastructure and deployment considerations
- Technical debt and migration concerns

### 5. Risk Coverage (0-100)

**Definition**: Are risks identified with mitigation strategies and ownership?

**Evaluation criteria**:
- Risks are specific, not generic ("Model hallucination in DCF calculations" vs. "AI accuracy issues")
- Each risk has a severity assessment
- Mitigation strategies are actionable
- Risk owners or responsible teams are identified
- Regulatory risks specific to the domain are covered
- Timeline risks related to dependencies

## Score Interpretation Guide

| Score Range | Interpretation |
|-------------|---------------|
| 90-100 | Production-ready — could be presented to stakeholders |
| 80-89 | Strong draft — needs minor refinement in weak dimensions |
| 70-79 | Working draft — specific dimensions need significant improvement |
| 60-69 | Early draft — useful structure but lacks depth |
| Below 60 | Needs regeneration with improved context injection |

## Cross-Agent Comparison

The scoring framework enables meaningful cross-agent comparison:

| Insight | What It Reveals |
|---------|----------------|
| Agent A scores higher on Risk Coverage than Agent B | Agent A's domain context includes more comprehensive risk frameworks |
| All agents score low on User Stories | The shared user story prompt template needs improvement |
| One agent's Technical Depth is an outlier | That domain may have more/fewer technical integration points |

This turns PRD generation from a "generate and hope" process into a **measurable, improvable system**.
